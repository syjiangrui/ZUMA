# HTML 移动端祖马项目方案

## 1. 项目目标

构建一个适合手机浏览器游玩的 HTML 祖马游戏原型，优先保证：

- 竖屏手机体验
- 核心玩法可玩
- 球体沿轨道移动稳定
- 球体具备可见纹理滚动效果
- 后续易于扩展关卡、特效、音效和 UI

当前阶段不追求商业级美术资源和完整内容量，先完成一个稳定、可迭代的技术底座。

## 2. 核心需求

### 必须实现

- 单页面 HTML 游戏
- `Canvas 2D` 渲染
- 一条固定曲线路径
- 球链沿路径推进
- 炮台瞄准与发射
- 发射球插入球链
- 三消与连锁消除
- 胜负判定
- 触屏操作可用
- 球体带纹理，并有明显滚动动画

### 后续扩展

- 多关卡配置
- 特殊球
- 音效与背景音乐
- 粒子特效
- 暂停/继续
- 本地最高分
- PWA 支持

## 3. 技术方案

### 技术栈

- `HTML`
- `CSS`
- `JavaScript`
- `Canvas 2D`

第一阶段先不引入框架和构建工具，直接用原生方式完成原型。原因：

- 原型开发速度快
- 调试简单
- 移动端运行链路短
- 便于先验证玩法和渲染结构

如果后续项目复杂度提升，再切换到 `Vite + 模块化 JavaScript` 或 `TypeScript`。

## 4. 核心架构

### 4.1 游戏模块划分

- `Game`: 游戏主循环、状态切换、全局更新和渲染
- `Path`: 路径定义、采样、按弧长取点
- `Chain`: 球链推进、插入、闭合、消除检测
- `ChainState`: 链段分裂、前段停驻、后段追赶、重新并链
- `Shooter`: 炮台朝向、当前球、下一球、发射逻辑
- `Projectile`: 飞行中的球
- `Renderer`: 背景、轨道、球体、UI 绘制
- `Input`: 触摸与鼠标统一输入
- `Assets`: 纹理生成或纹理加载

### 4.2 核心数据模型

轨道球的基础位置仍然由路径距离驱动，但不能只保留 `s`。

建议数据模型：

```js
{
  id: 17,
  paletteIndex: 2,
  radius: 18,
  s: 320,
  rotation: 0,
  offset: 0,
  offsetMode: "idle",
  impact: 0
}
```

说明：

- `s` 表示球当前的路径距离
- `rotation` 表示球纹理自转角
- `offset` 表示插入让位、消除补位等过渡位移
- `offsetMode` 用来区分正常、插入、补位等过渡状态
- `impact` 用于碰撞和接合的瞬时视觉反馈

除单球状态外，还需要一层链条状态：

```js
{
  chainHeadS: 680,
  splitState: {
    index: 12,
    frontOffset: 0
  }
}
```

说明：

- `chainHeadS` 表示整条链的基础推进距离
- `splitState` 表示当前链是否被中段消除切成两段
- 当存在 `splitState` 时，前半段和后半段不再共享完全相同的运动规则

飞行中的球使用另一套数据：

```js
{
  paletteIndex: 1,
  radius: 18,
  x: 240,
  y: 600,
  vx: 320,
  vy: -540,
  rotation: 0
}
```

## 5. 轨道与滚动动画方案

### 5.1 轨道表示

球链不直接依赖屏幕坐标速度，而是依赖轨道距离。

实现方式：

- 先定义一组控制点
- 用曲线或折线采样成大量 `pathPoints`
- 每个采样点存储 `x / y / len`
- `len` 表示从轨道起点到该点的累计长度

这样球的位置通过 `s` 映射到轨道点，而不是单独做二维速度积分。

### 5.2 球链推进

球链推进需要拆成三层：

- 基础推进层：整条链沿路径持续向终点推进
- 过渡层：插入让位、消除补位、局部冲击等临时位移
- 链段层：中段消除后，链条可以暂时分裂成前后两段

正常状态下：

- 头球每帧增加基础路径距离
- 后续球保持接近一个直径的逻辑间隔

插入时：

- 新球先以碰撞点附近的位置并入轨道
- 后半链条先让位，再逐步回到标准间距
- 三消判定不应在插入当帧立刻完成，应在插入动作基本成立后再触发

中段消除时：

- 前半段应先停住，不继续向终点推进
- 后半段单独追赶前半段，速度应由专门的补链规则控制
- 两段真正接触后，才能重新合并成一条链
- 重新合并后再做跨接缝的连锁消除判定

这部分不是表现优化，而是祖马核心玩法的一部分。

### 5.3 球体纹理滚动

要实现“看得见的滚动”，球必须带方向性纹理。

采用方案：

- 使用 Canvas 动态绘制球纹理
- 纹理包含条纹、旋涡、高光与暗部
- 球的自转角由路径位移驱动

核心公式：

```js
ball.rotation = ball.s / ball.radius;
```

说明：

- 这是二维视觉模拟中的“无滑动滚动”近似
- 球沿轨道前进越远，纹理旋转越多
- 纹理朝向和球沿路径移动方向分离处理

### 5.4 渲染方式

每帧渲染流程：

1. 通过 `s` 计算球心位置
2. 保存画布状态
3. 平移到球心
4. 旋转到 `rotation`
5. 绘制球纹理
6. 恢复画布状态

## 6. 移动端优先设计

### 6.1 屏幕与布局

- 优先竖屏
- 画布按视口自适应缩放
- 使用固定逻辑分辨率 + CSS 缩放方案

建议逻辑尺寸：

- `430 x 932` 附近的竖屏比例

### 6.2 输入

- 支持 `pointer` 事件
- 点击/拖动瞄准
- 抬手或点击触发发射
- 同时兼容桌面鼠标调试

### 6.3 性能

- 路径采样结果预计算
- 球纹理离屏缓存
- 避免每帧创建大量对象
- 控制粒子数量
- 尽量只用 `Canvas 2D` 基础能力，不上重滤镜

## 8. 第一阶段目录建议

建议先使用最小目录结构：

```text
game/
  index.html
  style.css
  main.js
```

在第一阶段中，先把逻辑写在 `main.js`，等原型稳定后再拆分模块。

## 9. 第一阶段关键技术决策

### 决策 1：不用 DOM 做球体动画

原因：

- 球数量多时 DOM 成本高
- 路径运动和旋转更适合 Canvas
- 更容易统一做粒子和特效

### 决策 2：先用程序化纹理，不先依赖外部图片

原因：

- 能快速验证滚动效果
- 减少资源依赖
- 后续可替换成正式贴图

### 决策 3：先做单关卡固定路径

原因：

- 先验证球链、插入和消除逻辑
- 降低第一阶段复杂度

## 10. 主要风险与应对

### 风险 1：插入与消除逻辑不稳定

应对：

- 明确区分轨道球和飞行球
- 所有轨道运动统一用 `s` 驱动
- 在 `s` 之外保留过渡层状态，不要把所有动画都硬塞进一个位置值
- 三消判定要和插入动画、并链动画解耦

### 风险 2：断链状态处理错误

应对：

- 把断链看成一个明确状态，而不是若干速度参数
- 前半段停驻、后半段追赶、重新并链要有独立规则
- 连锁判定不能跨越尚未闭合的断口

### 风险 3：滚动动画看起来不明显

应对：

- 纹理必须具备方向性
- 加强高光、暗纹和条纹对比
- 必要时叠加外圈装饰环

### 风险 4：移动端分辨率适配不稳

应对：

- 使用固定逻辑分辨率
- 用统一缩放策略映射到实际屏幕
- 先优先适配常见竖屏比例

### 风险 5：手感问题被误判为单纯数值问题

应对：

- 先判断是状态机问题还是参数问题
- 补位速度、停驻逻辑、并链时机要先正确，再做手感调参
- 不要试图只靠一个速度常量修复所有违和感

## 12. 当前建议

当前已经证明：项目不仅需要“路径推进”，还需要“链段状态管理”。

因此后续实现应遵守两个原则：

- 不再把整条球链视为始终完整的一段
- 不再把所有观感问题都当作速度常量问题

第一批代码目标应当是：

- 页面能在手机尺寸下正常显示
- 能看到一条轨道
- 能看到一串带纹理并持续滚动的球
- 能看到位于底部的炮台

当前阶段完成后，应优先验证以下逻辑正确性：

- 插入是否先发生让位再并入
- 中段消除后前半段是否停住
- 后半段是否以合理规则追上并重新接合
- 连锁判定是否只在真正接合后跨断口发生
