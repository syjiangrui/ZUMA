# HTML 移动端祖马项目方案

## 1. 项目目标

构建一个适合手机浏览器游玩的 HTML 祖马游戏原型，优先保证：

- 竖屏手机体验
- 核心玩法可玩
- 球体沿轨道移动稳定
- 球体具备可见纹理滚动效果
- 后续易于扩展关卡、特效、音效和 UI

当前阶段不追求商业级美术资源和完整内容量，先完成一个稳定、可迭代的技术底座。

## 2. 核心需求

### 必须实现

- 单页面 HTML 游戏
- `Canvas 2D` 渲染
- 一条固定曲线路径
- 球链沿路径推进
- 炮台瞄准与发射
- 发射球插入球链
- 三消与连锁消除
- 胜负判定
- 触屏操作可用
- 球体带纹理，并有明显滚动动画

### 后续扩展

- 多关卡配置
- 特殊球
- 音效与背景音乐
- 粒子特效
- 暂停/继续
- 本地最高分
- PWA 支持

## 3. 技术方案

### 技术栈

- `HTML`
- `CSS`
- `JavaScript`
- `Canvas 2D`

第一阶段先不引入框架和构建工具，直接用原生方式完成原型。原因：

- 原型开发速度快
- 调试简单
- 移动端运行链路短
- 便于先验证玩法和渲染结构

如果后续项目复杂度提升，再切换到 `Vite + 模块化 JavaScript` 或 `TypeScript`。

## 4. 核心架构

### 4.1 游戏模块划分

- `Game`: 游戏主循环、状态切换、全局更新和渲染
- `Path`: 路径定义、采样、按弧长取点
- `Chain`: 球链推进、插入、闭合、消除检测
- `ChainState`: 链段分裂、前段停驻、后段追赶、重新并链
- `Shooter`: 炮台朝向、当前球、下一球、发射逻辑
- `Projectile`: 飞行中的球
- `Renderer`: 背景、轨道、球体、UI 绘制
- `Input`: 触摸与鼠标统一输入
- `Assets`: 纹理生成或纹理加载

### 4.2 核心数据模型

轨道球的基础位置仍然由路径距离驱动，但不能只保留 `s`。

建议数据模型：

```js
{
  id: 17,
  paletteIndex: 2,
  radius: 18,
  s: 320,
  rotation: 0,
  offset: 0,
  offsetMode: "idle",
  impact: 0
}
```

说明：

- `s` 表示球当前的路径距离
- `rotation` 表示球纹理自转角
- `offset` 表示插入让位、消除补位等过渡位移
- `offsetMode` 用来区分正常、插入、补位等过渡状态
- `impact` 用于碰撞和接合的瞬时视觉反馈

除单球状态外，还需要一层链条状态：

```js
{
  chainHeadS: 680,
  splitState: {
    index: 12,
    frontOffset: 0
  }
}
```

说明：

- `chainHeadS` 表示整条链的基础推进距离
- `splitState` 表示当前链是否被中段消除切成两段
- 当存在 `splitState` 时，前半段和后半段不再共享完全相同的运动规则

飞行中的球使用另一套数据：

```js
{
  paletteIndex: 1,
  radius: 18,
  x: 240,
  y: 600,
  vx: 320,
  vy: -540,
  rotation: 0
}
```

## 5. 轨道与滚动动画方案

### 5.1 轨道表示

球链不直接依赖屏幕坐标速度，而是依赖轨道距离。

实现方式：

- 先定义一组控制点
- 用曲线或折线采样成大量 `pathPoints`
- 每个采样点存储 `x / y / len`
- `len` 表示从轨道起点到该点的累计长度

这样球的位置通过 `s` 映射到轨道点，而不是单独做二维速度积分。

### 5.2 球链推进

球链推进需要拆成三层：

- 基础推进层：整条链沿路径持续向终点推进
- 过渡层：插入让位、消除补位、局部冲击等临时位移
- 链段层：中段消除后，链条可以暂时分裂成前后两段

正常状态下：

- 头球每帧增加基础路径距离
- 后续球保持接近一个直径的逻辑间隔

插入时：

- 新球先以碰撞点附近的位置并入轨道
- 后半链条先让位，再逐步回到标准间距
- 三消判定不应在插入当帧立刻完成，应在插入动作基本成立后再触发

中段消除时：

- 前半段应先停住，不继续向终点推进
- 后半段单独追赶前半段，速度应由专门的补链规则控制
- 两段真正接触后，才能重新合并成一条链
- 重新合并后再做跨接缝的连锁消除判定

这部分不是表现优化，而是祖马核心玩法的一部分。

### 5.3 球体纹理滚动

要实现“看得见的滚动”，球必须带方向性纹理。

采用方案：

- 使用 Canvas 动态绘制球纹理
- 纹理包含条纹、旋涡、高光与暗部
- 球的自转角由路径位移驱动

核心公式：

```js
ball.rotation = ball.s / ball.radius;
```

说明：

- 这是二维视觉模拟中的“无滑动滚动”近似
- 球沿轨道前进越远，纹理旋转越多
- 纹理朝向和球沿路径移动方向分离处理

### 5.4 渲染方式

每帧渲染流程：

1. 通过 `s` 计算球心位置
2. 保存画布状态
3. 平移到球心
4. 旋转到 `rotation`
5. 绘制球纹理
6. 恢复画布状态

## 6. 移动端优先设计

### 6.1 屏幕与布局

- 优先竖屏
- 画布按视口自适应缩放
- 使用固定逻辑分辨率 + CSS 缩放方案

建议逻辑尺寸：

- `430 x 932` 附近的竖屏比例

### 6.2 输入

- 支持 `pointer` 事件
- 点击/拖动瞄准
- 抬手或点击触发发射
- 同时兼容桌面鼠标调试

### 6.3 性能

- 路径采样结果预计算
- 球纹理离屏缓存
- 避免每帧创建大量对象
- 控制粒子数量
- 尽量只用 `Canvas 2D` 基础能力，不上重滤镜

## 9. 第一阶段关键技术决策

### 决策 1：不用 DOM 做球体动画

原因：

- 球数量多时 DOM 成本高
- 路径运动和旋转更适合 Canvas
- 更容易统一做粒子和特效

### 决策 2：先用程序化纹理，不先依赖外部图片

原因：

- 能快速验证滚动效果
- 减少资源依赖
- 后续可替换成正式贴图

### 决策 3：先做单关卡固定路径

原因：

- 先验证球链、插入和消除逻辑
- 降低第一阶段复杂度

## 10. 主要风险与应对

### 风险 1：插入与消除逻辑不稳定

应对：

- 明确区分轨道球和飞行球
- 所有轨道运动统一用 `s` 驱动
- 在 `s` 之外保留过渡层状态，不要把所有动画都硬塞进一个位置值
- 三消判定要和插入动画、并链动画解耦

### 风险 2：断链状态处理错误

应对：

- 把断链看成一个明确状态，而不是若干速度参数
- 前半段停驻、后半段追赶、重新并链要有独立规则
- 连锁判定不能跨越尚未闭合的断口

### 风险 3：滚动动画看起来不明显

应对：

- 纹理必须具备方向性
- 加强高光、暗纹和条纹对比
- 必要时叠加外圈装饰环

### 风险 4：移动端分辨率适配不稳

应对：

- 使用固定逻辑分辨率
- 用统一缩放策略映射到实际屏幕
- 先优先适配常见竖屏比例

### 风险 5：手感问题被误判为单纯数值问题

应对：

- 先判断是状态机问题还是参数问题
- 补位速度、停驻逻辑、并链时机要先正确，再做手感调参
- 不要试图只靠一个速度常量修复所有违和感

## 11. Phase 2 执行顺序

当前 Phase 1 已具备：

- 中央发射口与环绕轨道
- 发射、插入与基础三消
- 中段消除后的基础断链与重新并链
- 纹理球滚动动画

因此 Phase 2 按以下顺序推进：

1. 建立 `gameState` 与单局生命周期
2. 接入正式失败/胜利判定
3. 补齐 HUD、结束提示与重开按钮，并把下一球预览纳入正式 UI
4. 建立分数、连击与连锁事件流
5. 完成正式的连锁消除规则
6. 在规则稳定后打磨断链吸附/回拉手感

## 12. 当前建议

当前原型已经证明：项目不仅需要“路径推进”，还需要“链段状态管理”和“单局状态闭环”。

因此 Phase 2 应遵守三个原则：

- 先补状态闭环，再补手感
- 计分、连击和连锁不要散落在渲染逻辑里，应由消除事件驱动
- 断链回拉的优化要建立在胜负、重开和连锁判定已经稳定的前提上

Phase 2 的成功标准应当是：

- 玩家能完整开始、进行并结束一局游戏
- 每次消除和连锁都有明确反馈
- 断链闭合与跨缝连锁行为可预测
- 出现问题时能通过重开快速复现与调试

在 Phase 2 中，应优先验证以下逻辑正确性：

- `playing / win / lose` 是否正确控制更新循环
- 胜负判定是否不会与连锁判定互相打架
- 分数与连击是否来自同一套消除事件，而不是从 UI 反推
- 断链重新接合后，是否只在真实闭合后触发跨缝连锁

## 13. Phase 2 详细任务拆解

### 13.1 目标与边界

Phase 2 的核心目标不是继续堆原型功能，而是把当前系统补成“可以完整打一局”的结构。

纳入范围：

- 单局状态机
- 胜负判定
- 重开流程
- HUD 与结束反馈
- 分数、连击、连锁统计
- 正式连锁消除规则
- 断链吸附/回拉手感打磨

暂不纳入：

- 音效与粒子
- 多关卡与特殊球
- 本地存档
- 复杂暂停系统

### 13.2 工作包 A：单局状态机与生命周期

目标：把当前“原型循环”升级成“有开始、进行、结束、重开”的回合生命周期。

任务：

- 引入统一的 `gameState`
- 明确 `playing / win / lose` 三种主状态
- 让链条推进、发射球更新、碰撞检测只在 `playing` 下运行
- 提供统一的重置入口，复用当前 `createChain()`、发射器重置和临时状态清理逻辑

验收标准：

- 游戏结束后球链、发射球和输入不会继续推进
- 不刷新页面也能完整重开一局
- 不会出现上一局的 `splitState`、`pendingMatchChecks` 或发射球残留到下一局

说明：

- `paused` 不是当前阻塞项，可以先不纳入 Phase 2

### 13.3 工作包 B：失败/胜利判定与回合出口

目标：把当前原型里“越线后直接重建球链”的行为，升级成正式的胜负结果。

任务：

- 把当前 `tailS > totalPathLength + EXIT_GAP` 的原型重置逻辑改为 `lose`
- 定义 `win` 条件：球链清空，且没有飞行球、没有待处理连锁检查
- 区分“胜负判定成立”和“真正进入结束状态”的时机，避免和插入/连锁同帧冲突

验收标准：

- 球链越线后进入失败状态，而不是立即开始新局
- 最后一组球消失时，不会因为还有飞行球或待结算连锁而误判胜利
- 胜负状态能稳定显示到玩家主动重开

说明：

- 第一版可沿用当前原型的越线语义，先把结果状态做正确；是否进一步改成更像原作的吞口判定，可留到后续微调

### 13.4 工作包 C：HUD、下一球预览与重开入口

目标：把已有的原型信息显示整理成正式 HUD，并补齐结束 UI。

任务：

- 整理顶部信息区，至少包含分数、连击和当前状态提示
- 将下一球预览从“原型可见元素”升级为正式 HUD 组成部分
- 提供可点击/可触控的重开按钮
- 增加 `win / lose` 的结束提示层

验收标准：

- 玩家无需看控制台即可理解当前状态
- 下一球预览在整局中始终与实际出球一致
- 结束后可通过按钮快速重新开始

### 13.5 工作包 D：分数、连击与事件模型

目标：不要把计分写成若干临时加法，而是建立稳定的玩法事件流。

任务：

- 为每次发射建立一次“行动上下文”，用于归并后续消除与连锁
- 为一次消除生成统一事件，至少包含消除数量、触发来源、是否跨缝、所属行动
- 由消除事件驱动分数增加、连击层数和 HUD 提示
- 明确“基础消除分”和“连锁加成分”的计算规则

验收标准：

- 分数变化只来自消除事件，不从 `chain.length` 差值反推
- 同一发球引发的连续消除能累积为连击
- 新的一发球会重置上一轮的连击上下文

说明：

- 这一步的关键不是 UI，而是先把“计分依据是什么”定义清楚

### 13.6 工作包 E：正式连锁消除规则

目标：把当前基础三消升级成完整的“首消 -> 断链 -> 重新接合 -> 再判定”的规则链。

任务：

- 明确一次发射后的首轮消除判定
- 明确断链重新接合后的二次判定
- 只允许在真实闭合后跨断口触发连锁
- 将连锁次数和工作包 D 的事件模型打通

验收标准：

- 中段消除后，两段未接上时不会跨缝误判
- 两段接合后如果颜色成立，会触发第二轮及后续连锁
- 连锁次数与实际发生的消除轮次一致

### 13.7 工作包 F：断链吸附/回拉手感打磨

目标：在规则已经稳定的前提下，把断链后的追赶与闭合做得更像祖马。

任务：

- 重新审视后半段追赶速度，避免明显快于正常滚动速度
- 调整接缝闭合阈值与吸附时机，减少“突然吸上去”的感觉
- 必要时为接缝闭合增加轻微冲击或回弹，但不改变底层判定
- 把这一层优化限制在 `splitState` 及其相关过渡，不再引入新的大状态机实验

验收标准：

- 前半段停驻、后半段追赶、重新并链三个阶段清晰可感
- 后半段不会在接合前表现出明显失真的超速
- 闭合后连锁判定仍然稳定，不因手感优化引入新 bug

### 13.8 推荐实施顺序

推荐按以下工作包顺序执行：

1. 工作包 A：先补生命周期
2. 工作包 B：再补胜负出口
3. 工作包 C：让调试和试玩有正式入口
4. 工作包 D：建立计分事件流
5. 工作包 E：完成正式连锁
6. 工作包 F：最后打磨断链手感

原因：

- A 到 C 先解决“一局是否成立”
- D 和 E 再解决“奖励反馈是否正确”
- F 最后才处理“手感是否像原作”
