# Zuma 路径平滑方案分析

> 更新日期: 2026-04-21
> 关联文件: `path.js`, `path-editor.html`, `path-fit.js`, `render.js`

## 背景

当前项目的路径系统已经分成两层：

- **游戏层**：所有球的运动、碰撞、插入、匹配，都是沿着 `pathPoints[]` 这条离散采样路径走的。
- **渲染层**：屏幕上画出来的轨道，是 `cachedTrackPath` 这条 `Path2D`。

也就是说，**视觉轨道怎么画**，和**球怎么运动**，本来就可以分开设计。

这点很关键，因为“看起来不够平滑”通常是渲染问题，不一定要改动运动系统。

## 当前实现

### 1. Bezier 关卡的数据来源

编辑器保存的是二次贝塞尔链：

- 每段 3 个点：`p1, cp, p2`
- 游戏启动时由 `levels.js` 展平成 `pathParams.points`
- `path.js` 的 `generateBezierPath()` 再把每段二次贝塞尔采样成密集点

### 2. 运动轨迹如何计算

游戏里的球并不是沿原始贝塞尔方程实时求点，而是沿着已经采样好的 `pathPoints[]` 运动：

- 先把整条路径采样成很多离散点
- 给每个点累计弧长 `len`
- 球的位置用 `s` 表示路径距离
- 运行时通过 `getPointAtDistance()` 在邻近采样点之间线性插值

这套方案是对的，不建议轻易推翻。它对祖玛这类“沿轨道排序 + 插入 + 匹配”的玩法很稳定。

### 3. 当前视觉为什么会不够顺

核心问题通常有两个：

1. **拟合后贝塞尔段之间接头不够平滑**。
原因不是“Bezier 天生不平滑”，而是“多段二次贝塞尔如果控制点关系不好，就只有位置连续，没有切线连续”。

2. **视觉轨道不是直接按作者定义的曲线画，而是经过中间表示再画**。
如果渲染走的是“采样点 -> 再插值”的链条，局部会有二次误差累积。

因此要先分清：

- **Bezier 本身的问题**：段与段的 G1/C1 连续性处理不好
- **采样再渲染的问题**：显示出来的轨道不是原始曲线，而是一个近似版本

## 先回答结论

如果问题是：

> “现在插值 Bezier 之后不够平滑，按这个问题该选什么方案？”

我的结论是：

### 推荐方案

**保留游戏层的弧长采样运动不变；渲染层直接画原始曲线；编辑层优先用 waypoint 骨架；如果后续还要进一步升级，再考虑切到 cubic B-spline / centripetal Catmull-Rom。**

换句话说，优先级应该是：

1. **先把渲染和运动解耦彻底**
2. **视觉轨道直接用原始曲线画，不走二次采样后的近似线**
3. **接头处做强制平滑或换更高阶曲线表示**
4. **只有在二次贝塞尔链仍然难维护时，再整体迁移到 B-spline**

这比直接“把系统全改成 B-spline”更稳，也更符合当前代码结构。

## Bezier 和 B-Spline 的对比

### 方案 A：继续用二次 Bezier

优点：

- 已经和当前编辑器、存档、关卡文件结构兼容
- Canvas 原生支持 `quadraticCurveTo`，渲染成本低
- 运动层已经基于它的采样结果工作，改动最小
- 入口、终点、关键控制点都能精确指定

缺点：

- 多段拼接时很容易出现接头不平滑
- 二次贝塞尔自由度偏低，复杂弯道往往要拆更多段
- 自动拟合时如果段数过多，视觉质量高度依赖接头处理

适用条件：

- 你希望**最小改动**解决问题
- 你已经有一批贝塞尔关卡数据
- 你愿意在拟合后加一层接头平滑

### 方案 B：换成三次 Bezier 链

优点：

- 比二次贝塞尔多一个自由度，拟合更自然
- 实现 G1/C1 平滑更容易
- 仍然是常见、好调、好画、好导出的表示

缺点：

- 数据结构要从 3 点一段改成 4 点一段
- 编辑器与导出逻辑都要跟着改
- 虽然比二次更顺，但本质上还是“分段曲线链”，接头管理仍然存在

适用条件：

- 你愿意接受一次中等规模重构
- 目标仍然是“可控、明确、能精确过点/控切线”

### 方案 C：换成 B-Spline

优点：

- 天然更平滑，尤其是 cubic B-spline 很容易得到连续二阶导数
- 作者只摆 waypoint，不必直接操心控制柄
- 对连续大弯和 S 弯通常比二次贝塞尔链更自然

缺点：

- **B-spline 通常不穿过控制点**，而是“被控制点拉住”
- 端点、入口、终点需要做 clamped / endpoint duplication 处理
- 想保留祖玛路径里那种“明确拐一下再进下一段”的形状时，反而会被它过度抹平
- 如果某个角需要更硬，就必须引入重复 knot、局部断开、corner tag 等额外机制
- 现有编辑器和存档格式基本要重新定义

适用条件：

- 你准备把路径系统改成**纯 waypoint 驱动**
- 你接受“作者输入的是骨架，不是精确过点曲线”
- 你愿意实现 clamped spline、corner 标记、按段打断等配套逻辑

### 方案 D：Centripetal Catmull-Rom

它不是 B-spline，但在体验上经常是一个更现实的折中。

优点：

- 通常**通过 waypoint 本身**，比 B-spline 更符合“我点哪条路就走哪条路”的直觉
- 比普通折线、二次贝塞尔链更平滑
- 可以再转换成 cubic Bézier 供 Canvas 绘制

缺点：

- 想保留硬拐角时，仍然要做分段/断开
- 仍需要实现额外一层 spline 求值或转换逻辑

适用条件：

- 你更关心“骨架点就是路径必经点”
- 想要比 B-spline 更直观的编辑手感

## 针对“现在不够平滑”最该先做什么

### 最优先，不是直接换 B-spline

如果当前痛点只是：

- 一键拟合后局部接头有折感
- 轨道看起来不像编辑器里那条线那么顺

那么**第一优先级不是换算法家族，而是把当前链条中的近似误差去掉**。

应先做：

1. **渲染轨道直接按原始曲线画**
2. **运动仍然沿采样弧长点走**
3. **拟合产出的相邻段强制做 G1/C1 平滑**

这一步完成后，通常视觉质量会明显提升，而且不会破坏玩法层稳定性。

## 为什么不建议现在直接切 B-Spline

从当前项目状态看，不建议立刻全量切到 B-spline，原因很实际：

### 1. 当前问题不是“采样运动错误”

球的运动、碰撞、路径距离系统本身没有明显架构问题。

问题主要是：

- 拟合链的连续性
- 渲染不是直接使用最原始曲线

这两个都能在现有框架里修掉。

### 2. B-spline 会引入新的编辑语义

切到 B-spline 后，作者心智会从：

- “我定义这段曲线长这样”

变成：

- “我放这些控制点，让曲线大概这样经过”

这对工具交互是一次本质变化，不只是底层函数替换。

### 3. 祖玛路径不一定希望全局太顺

祖玛路径通常不是赛车轨道那种“越顺越好”，而是需要：

- 清晰可读的走向
- 某些刻意设计的压迫性弯道
- 一些视觉上明确的节奏点

纯 B-spline 往往会把这些都软化掉，除非你再补 corner 系统。

## 什么时候 B-Spline 值得上

如果未来决定彻底升级成：

- **只编辑 waypoint，不再直接编辑贝塞尔控制点**
- **所有路径都由统一 spline 内核生成**
- **允许对点打标签：smooth / corner / break**
- **渲染和采样都从同一 spline evaluator 来**

那时 B-spline 或 centripetal Catmull-Rom 就值得认真上。

在这种模式下，推荐顺序反而是：

1. waypoint 骨架
2. corner/break 标记
3. clamped cubic spline 求值
4. 采样成 `pathPoints[]` 给游戏层
5. 渲染直接画 spline 或转换后的 cubic Bézier

## 当前项目的推荐落地方案

### 短期推荐

**保留“waypoint 编辑 + 一键拟合 + 游戏层采样运动”这条主线，不切 B-spline。**

具体做法：

1. waypoint 只负责定义骨架和关键转折
2. 拟合结果继续落成贝塞尔曲线，便于存档与调试
3. 游戏里的球继续沿 `pathPoints[]` 运动
4. 屏幕上轨道直接按原始曲线画，不再靠二次近似重建
5. 对自动拟合出的接头统一做平滑后处理

这是当前工程成本最低、收益最高的方案。

### 中期推荐

如果二次贝塞尔链仍然经常需要很多段、很多补丁，下一步优先考虑：

**二次 Bezier → 三次 Bezier**

而不是直接一步跳到 B-spline。

理由：

- 更平滑
- 表达能力更强
- 仍然保留“显式曲线段”的工程可控性
- 迁移成本明显小于“整套 spline 语义重写”

### 长期推荐

如果最终目标是“像关卡编辑器一样只拉骨架点，不碰控制柄”，再考虑：

**waypoint + cubic B-spline / centripetal Catmull-Rom + corner tag**

这会是更干净的第二代路径系统。

## 最终建议

针对当前这个项目，我建议按下面的判断执行：

- **如果只是现在看起来不平滑**：不要先换 B-spline，先把渲染改成直接画原始曲线，并修好接头连续性。
- **如果后续发现二次贝塞尔表达力本身不足**：优先升级到三次贝塞尔链。
- **如果未来想彻底改成 waypoint-only 的骨架编辑器**：再考虑 clamped cubic B-spline 或 centripetal Catmull-Rom。

一句话总结：

**当前问题更像“渲染和接头连续性问题”，不是“必须改成 B-spline 才能解决的问题”。**
